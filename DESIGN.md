# 建筑模型合规性检查 Agent —— 技术设计文档

## 1. 项目目标

构建一个智能 Agent，输入建筑设计模型（IFC / 简化 JSON 等），自动检查若干条基础合规与合理性规则，输出结构化的检查报告（含问题定位、严重程度、修改建议），并提供可视化/可读的展示形式。

核心考察点：**可运行 + 有思考 + 有品味**。因此本项目策略是：
- **少而精**：只做 2 条规则，但做透（数据解析、规则判断、报告生成、可视化全链路打通）。
- **工程判断力优先于建筑知识**：选择规则时优先选"规则逻辑清晰、可验证、数据易得"的，而不是最"高大上"的碰撞检测（碰撞检测几何计算复杂，投入产出比低）。

---

## 2. 规则选型与理由

调研了常见 BIM 合规检查方向：几何碰撞、疏散门净宽、房间到出口距离、构件属性完整性（FireRating/名称/材质）、房间面积/用途合理性等。

**最终选定 2 条规则：**

### 规则 1：疏散门/走道净宽度检查（消防规范类）
- **依据**：参考《建筑设计防火规范》GB 50016，疏散门净宽通常要求 ≥ 0.9m（单扇门），高层住宅疏散走道净宽 ≥ 1.1m～1.4m（依据建筑类型不同）。为简化，Agent 中做成**可配置阈值**（如门 ≥ 0.9m，走道 ≥ 1.2m），并在报告中注明"示例阈值，实际需按项目所在地规范核实"。
- **理由**：规则逻辑简单（数值比较），且是最容易在 IFC 中直接拿到的几何属性（`IfcDoor`/`IfcOpeningElement` 的 `OverallWidth`），工程可解释性强，业主/设计师一看就懂。
- **判断输出**：门/走道 ID、宽度实测值、规范阈值、是否合规、差值。

### 规则 2：构件关键属性完整性检查（模型质量类）
- **依据**：BIM 交付标准（如 Uniclass / COBie / 国内 BIM 施工图交付标准）通常要求门窗、防火分区构件必须填写 `FireRating`（耐火等级）、`Name`（构件名称）等关键属性，缺失会导致后续消防审查、算量出错。
- **理由**：这是"模型可用性"层面的合理性检查，实现成本低（属性是否为空的判断），但在真实工程中极其常见且有价值（很多模型问题不是几何错误，而是属性缺失导致下游流程卡住）。
- **判断输出**：构件 ID/类型、缺失的属性列表、严重程度（如 `FireRating` 缺失在防火门上标记为高优先级）。

> 备选但暂不做的规则：几何碰撞检测（墙/梁/管道）——需要三维包围盒/网格相交算法，实现和调试成本较高，超出"1-2条规则、精炼优先"的定位，作为后续可扩展方向在文档中说明。

## 3. 数据格式与测试数据

### 3.1 输入格式
优先支持两种输入，兼顾"真实感"与"易实现"：

1. **简化 JSON 模型**（主要测试数据，自行构造，模拟从 IFC 提取后的结构）：
```json
{
  "project": "Demo Building A",
  "doors": [
    {"id": "D-101", "type": "IfcDoor", "name": "疏散门-1F-东", "overallWidth": 0.85, "fireRating": "", "isEscapeDoor": true},
    {"id": "D-102", "type": "IfcDoor", "name": "", "overallWidth": 1.0, "fireRating": "A1.0h", "isEscapeDoor": true}
  ],
  "corridors": [
    {"id": "C-01", "name": "1F走道", "width": 1.35, "isEscapeRoute": true}
  ]
}
```
   - 好处：结构清晰、可控，便于快速构造"必然触发规则"的测试用例（如故意设一扇 0.85m 的疏散门）。

2. **真实 IFC 文件（.ifc）**（进阶/加分项，用 `ifcopenshell` 解析）：
   - 从 buildingSMART 官方示例库或 IfcOpenShell 自带的 sample IFC 文件中取一个小型建筑模型（如 `Duplex_A_20110907.ifc`，业界常用的公开测试模型）。
   - 用 `ifcopenshell` 提取 `IfcDoor.OverallWidth`、`Pset_DoorCommon.FireRating` 等信息，转换成与上面 JSON 相同的内部数据结构（统一后续规则引擎的输入接口）。

> 架构上采用"适配器模式"：IFC 解析器 / JSON 解析器 → 统一的内部数据模型（Pydantic/TypeScript interface）→ 规则引擎。这样保证无论输入哪种格式，规则代码只写一份。

## 4. 技术架构

```
┌─────────────┐      ┌──────────────┐      ┌───────────────┐      ┌───────────────┐
│  输入文件    │ ---> │  数据解析层   │ ---> │   规则引擎     │ ---> │  报告/可视化层  │
│ .ifc / .json│      │(ifcopenshell/│      │ (Rule 1, 2...) │      │(终端/HTML/图表) │
└─────────────┘      │ json loader) │      └───────────────┘      └───────────────┘
                      └──────────────┘
                              │
                        统一内部数据模型
                    (Door / Corridor / Element)
```

### 技术栈选择
- **语言**：Python（生态最成熟，`ifcopenshell` 是 IFC 解析事实标准；数据处理、报告生成都方便）。
- **IFC 解析**：`ifcopenshell`
- **数据模型**：`pydantic`（类型校验 + 清晰的 schema，体现工程规范性）
- **规则引擎**：轻量自研（一个 `Rule` 抽象基类 + 每条规则一个子类，`check(model) -> list[Issue]`），不引入过重的规则引擎框架，保持"精炼"。
- **报告输出**：
  - 终端彩色文本摘要（Rich 库，快速看结果）
  - HTML 报告（含表格 + 简单图表，直观展示违规构件、严重程度分布），用于演示时展示"有品味"
- **交互形式**：命令行工具（CLI），支持 `python check.py --input model.ifc` 或 `--input model.json`，输出报告文件。如果时间允许，用 Streamlit 做一个极简 Web 界面（上传文件 → 一键检查 → 可视化结果），提升演示效果。
- **Agent 化设计**（体现"智能 Agent"而非单纯脚本）：
  - 用一个轻量的 LLM（如通过 OpenAI/Claude API 或本地规则+LLM解释混合）在规则判断之外，加入"自然语言解读"环节：Agent 读取规则引擎输出的结构化 Issue 列表，生成一段人类可读的**总结与整改建议**（如"发现2处疏散门宽度不足，建议优先处理D-101，因其位于主要疏散路径..."）。
  - 这一步是"规则引擎 + LLM 总结"的组合，既保证结果确定性（规则硬编码判断，不依赖 LLM 产生幻觉数值），又体现 Agent 的智能交互和解释能力。
  - Prompt 设计会单独保存在仓库的 `prompts/` 目录下，作为提交内容的一部分。

## 6. 报告输出设计（体现"人机交互友好、可视化有帮助"）

每条 Issue 包含：
```
{
  "rule_id": "R1-EscapeWidth",
  "severity": "high" | "medium" | "low",
  "element_id": "D-101",
  "element_name": "疏散门-1F-东",
  "message": "疏散门净宽 0.85m 低于规范要求 0.90m",
  "measured_value": 0.85,
  "threshold": 0.9,
  "suggestion": "建议调整门洞尺寸或更换为标准宽度疏散门"
}
```

HTML 报告结构：
1. 顶部摘要卡片：总检查项数 / 通过数 / 违规数（按严重程度分色）
2. 表格：逐条列出违规项，可按规则/严重程度筛选
3. 底部：LLM 生成的自然语言总结与优先级建议
4. （加分）用简单柱状图展示"各规则违规数量分布"

## 7. 测试计划

- 构造至少 3 个测试场景：
  1. **全部合规**的模型（验证无误报）
  2. **规则1触发**（疏散门/走道宽度不足）
  3. **规则2触发**（属性缺失，尤其防火门缺 FireRating）
- 单元测试覆盖规则判断的边界值（如恰好等于阈值、略高于/略低于阈值）。
- 若使用真实 IFC 样例，验证解析层能正确提取字段，且在字段缺失时不崩溃（容错处理）。
